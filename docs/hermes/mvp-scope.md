## Product goal

Build a simple local web app or desktop-style UI for a small professional services team handling client financial and tax documents. The MVP helps the user locally detect, review, pseudonymise, and export redacted documents before sharing or downstream processing.

## Non-negotiable security rule

- Development may use synthetic/fake data only.
- Real client files must be processed locally only.
- Real client files, extracted text, screenshots, mappings, audit logs, or document contents must not be sent to Telegram, Codex, OpenAI, Claude, or any public AI service.
- Any sample data needed by workers must be synthetic and created specifically for testing.
- The project `inputs/` directory is approved synthetic data for development, so Codex workers may build and test against it.
- Replacement maps and audit logs are stored locally only.

## Primary MVP user

Amelie / a small professional services team handling client financial and tax documents.

User capability: semi-technical, but the app should feel simple and not require command-line use for normal operation.

## First supported workflow

MVP v0.1 supports PDF first.

Required flow:

1. Choose local PDF files from a local folder.
2. Choose or create a local redaction profile for the job/client/matter.
3. Add known sensitive custom terms locally, such as client names, person names, company/trust names, addresses, investor IDs, aliases, spelling variants, and other terms the user wants pseudonymised.
4. Extract text locally.
5. Detect likely PII locally using custom terms, deterministic patterns, context/label rules, and optional local-only suggestions.
6. Show a review screen with detections.
7. Allow the user to approve, reject, and edit detections before export.
8. Export redacted copies.
9. Produce a local audit log and local replacement map.

## Next file types after PDF

After PDF workflow is usable and reviewed:

1. DOCX.
2. Excel / CSV.

Images/scans are not part of the first locked workflow unless needed to support scanned PDFs later.

## PII/entity detection policy

MVP must detect likely:

- Person names.
- Addresses.
- Emails.
- Phone numbers.
- Account numbers.
- Investor/client IDs.
- TFN/ABN-like numbers.
- Dates of birth, where present.
- Company and trust names.
- User-defined custom terms.

## Local custom redaction profiles

Custom redaction profiles are a core MVP feature.

The app must let the user create/select a local profile containing known sensitive terms for a job/client/matter. Profile terms may include:

- Person names.
- Company names.
- Trust names.
- Addresses.
- Account/investor/client identifiers.
- Adviser or related-party names.
- Abbreviations, aliases, and spelling/capitalisation variants.
- Other custom terms the user decides should be pseudonymised.

Profile entries should include:

- Original term.
- Entity type.
- Replacement label, either user-provided or generated by the app.
- Optional variants.
- Optional notes.

Profiles must be stored locally only. They must not be sent to Telegram, Codex, OpenAI, Claude, or public AI. Worker prompts and progress reports must avoid quoting profile contents unless the contents are confirmed synthetic test data.

The app may locally suggest candidate terms from parsed synthetic/local documents, but suggested terms must enter the review screen as pending items and must not be auto-exported without user review.

## Redaction/pseudonym policy

- MVP uses human-reviewed pseudonym replacement.
- Replacement labels should follow stable typed placeholders, for example:
  - `[PERSON_1]`
  - `[COMPANY_1]`
  - `[TRUST_1]`
  - `[ACCOUNT_1]`
  - `[CLIENT_ID_1]`
  - `[ADDRESS_1]`
  - `[EMAIL_1]`
  - `[PHONE_1]`
  - `[DOB_1]`
  - `[ABN_1]`
  - `[TFN_1]`
  - `[CUSTOM_1]`
- The user must be able to approve/edit detections before redaction is exported.
- Pseudonym mappings must remain local only.

## PDF redaction requirement

For PDFs, the MVP should use true redaction or a safe generated output where possible. It must not rely only on visual cover boxes that leave underlying PII recoverable.

Acceptance for PDF export must include a check that sensitive source strings are not extractable from the redacted output for the synthetic test fixture.

## Human review checkpoints

Before export, the user must be able to:

- Review detected entities in context.
- Edit entity type and replacement label.
- Add missed detections/custom terms.
- Create, edit, import, and reuse local custom redaction profiles.
- Reject false positives.
- Confirm the final replacement map before export.

## Audit and mapping outputs

Local-only outputs:

- Redacted copy/copies.
- Replacement map.
- Audit log containing input file name/path reference, processing timestamp, entity categories, reviewer actions, and export status.

Audit logs should avoid storing full original PII where possible; prefer local references, entity type, page/position/context metadata, and replacement label.

## MVP non-goals

- Public/cloud AI processing of real client documents.
- Sending real document text or screenshots to Telegram, Codex, OpenAI, Claude, or public AI.
- Command-line-only user workflow.
- Perfect automated detection without human review.
- Batch automation without a review gate.
- DOCX/Excel/CSV implementation before the PDF path is usable.
- Final production UI architecture decisions beyond keeping business logic reusable outside the UI shell.

## Implementation consequences

- Keep extraction, detection, review state, redaction/export, audit logging, and mapping logic in reusable local services/modules, not buried in the UI layer.
- Implement local custom redaction profiles as a first-class service/model, not as one-off YAML replacements buried in the old CLI config.
- The detector pipeline should include CustomTermDetector, RegexDetector, ContextLabelDetector, optional PresidioDetector, and optional LocalLLMDetector, all producing the same DetectionCandidate model for review.
- Use synthetic fixtures for all tests and worker tasks.
- Add guardrails/docs so workers do not inspect, upload, or quote real local files from `outputs/` or similar folders.
- Build PDF first with tests proving exported redacted PDFs do not expose synthetic source PII through text extraction.
- Treat the UI as a simple local MVP shell that can later be replaced.
